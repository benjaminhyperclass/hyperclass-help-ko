# RFC 5322 비준수 이메일 메시지: 완전 문제 해결 가이드

## 무슨 일이 일어나고 있나요?

고객님의 이메일이 RFC 5322(이메일 형식을 규정하는 공식 기준)에서 요구하는 기술적 형식 표준을 충족하지 않아 수신 서버에서 거부되고 있습니다. 이메일 헤더, 발신자 주소, 또는 메시지 구조가 확립된 이메일 프로토콜을 위반할 때 이러한 거부가 발생하며, 수신 서버가 전송을 완전히 거부하게 됩니다.

## 빠른 진단: RFC 5322 준수 문제 확인하기

### 반송 메시지 예시

- "From 헤더나 도메인이 필수 기술 표준을 충족하지 않아 이메일이 거부되었습니다"
- "프로토콜 또는 형식 문제로 인해 메시지가 거부되어 연결이 거부되었습니다"
- "메일 루프나 과도한 홉으로 인해 메시지가 거부되어 이메일 전송 표준을 위반했습니다"
- "발신자 주소나 메시지 형식이 이메일 표준을 충족하지 않거나 수신 서버에서 허용되지 않습니다"
- "이메일 메시지가 필수 형식 또는 헤더 표준(누락되거나 잘못된 From 헤더 또는 허용되지 않는 문자 등)을 충족하지 않습니다"
- "콘텐츠나 첨부 파일이 Gmail의 보안 또는 콘텐츠 가이드라인을 위반하여 메시지가 차단되었습니다"
- "이메일 메시지 형식이나 헤더가 필수 표준을 준수하지 않습니다"
- "발신자 주소 형식이 이메일 표준(RFC 5321/5322)을 준수하지 않습니다"

## RFC 5322 준수 이해하기

### 핵심 개념

RFC 5322에서 정의하는 내용:

- 적절한 이메일 헤더 구조 및 형식 요구사항
- 유효한 발신자 주소 형식 및 문자 제한
- 메시지 라우팅 프로토콜 및 홉 제한
- 이메일 메시지의 필수 및 선택적 헤더 필드
- 국제 콘텐츠를 위한 문자 인코딩 표준

### 일반적인 준수 위반 사항

헤더 문제:
- From 헤더 누락 또는 잘못된 형식
- 발신자 이름이나 주소의 잘못된 문자
- 헤더의 부적절한 날짜 형식
- 과도하거나 잘못된 형식의 Received 헤더

주소 형식 문제:
- 특수 문자가 제대로 인코딩되지 않음
- 이메일 주소 주변의 꺾쇠괄호 누락
- 잘못된 도메인 이름 형식
- 표시 이름의 부적절한 사용

메시지 구조 문제:
- 과도한 홉을 유발하는 메일 루프
- 라인 길이 위반 (998자 초과)
- 필수 메시지 경계 누락

## RFC 5322 준수 문제 단계별 해결

### 1단계: 모든 발신자 설정 점검하기

**지시사항:**

- **캠페인 발신자 설정 검토:**
  `Marketing(마케팅)` → `Email(이메일)` → `Campaigns(캠페인)`으로 이동
- 전송 문제가 있는 각 캠페인을 열어주세요
- 캠페인 내의 이메일 단계를 클릭하세요
- "From Name"과 "From Email" 필드가 적절한 형식인지 검토하세요

- **워크플로우 이메일 액션 확인:**
  `Automation(자동화)` → `Workflows(워크플로우)`로 이동
- 이메일 액션이 포함된 워크플로우를 열어주세요
- 워크플로우의 각 "Send Email" 액션을 클릭하세요
- 발신자 이름과 이메일 주소 설정을 검토하세요

- **일괄 작업 이메일 설정 점검:**
  `Contacts(연락처)` → `Bulk Actions(일괄 작업)`로 이동
- 일괄 작업 설정의 발신자 이름과 이메일 필드를 검토하세요
- 인증된 발신 도메인과의 일치성을 확인하세요
- 일괄 이메일 발신자 정보가 캠페인 표준과 일치하는지 확인하세요

- **모든 발신자 정보 표준화:**
  발신자 이름에는 영숫자, 공백, 기본 구두점만 포함하도록 하세요
- 발신자 이름에서 따옴표, 대괄호, 기호 등의 특수 문자를 제거하세요
- 이메일 주소가 name@domain.com 형식을 따르고 특수 문자가 없는지 확인하세요
- 표시 이름을 64자 이하로 유지하세요
- 모든 이메일 접점에서 일관된 발신자 이름 형식을 사용하세요
- 모든 발신자 이메일 주소가 제대로 인증되었는지 확인하세요
- 회신 주소가 유효하고 적절한 형식인지 확인하세요

#### 발신자 설정 성공

- 모든 캠페인 발신자 주소가 적절한 이메일 형식 표준을 따름
- 워크플로우 이메일 액션이 일관되고 준수하는 발신자 정보를 사용함
- 일괄 작업 이메일이 적절한 형식과 구조를 유지함
- 모든 이메일 접점에서 표시 이름이 표준 문자만 포함함

### 2단계: 이메일 템플릿 구조 검증하기

**지시사항:**

- **이메일 템플릿 접근:**
  `Marketing(마케팅)` → `Emails(이메일)` → `Templates(템플릿)`으로 이동
- 전송 문제가 있는 템플릿을 열어주세요
- 템플릿 준수 문제 수정:
  라인 길이가 998자를 초과하지 않는지 확인하세요
- 다양한 이메일 클라이언트에서 템플릿을 테스트하세요

### 3단계: 외부 검증 도구 사용하기

- **Mail-Tester.com:**
  캠페인, 워크플로우, 또는 일괄 작업에서 제공된 주소로 테스트 이메일을 보내세요
- 상세한 준수 리포트를 검토하세요
- 식별된 RFC 5322 위반 사항을 해결하세요

- **MXToolbox 이메일 헤더 분석기:**
  전송된 메시지의 이메일 헤더를 복사하세요
- 헤더 분석기 도구에 붙여넣으세요
- 준수 경고 및 오류를 검토하세요

- **RFC 5322 검증 도구:**
  온라인 RFC 준수 검사기를 사용하세요
- 연락처를 가져오기 전에 이메일 주소 형식을 검증하세요
- 표준 준수를 위한 메시지 구조를 테스트하세요

## 복구 일정 및 예상 결과

### 1단계: 즉시 수정 (1-2시간)

- **조치:** 캠페인, 워크플로우, 일괄 작업의 발신자 주소 형식 수정

- **조치:** 모든 이메일 템플릿의 적절한 구조 검토 및 수정
- **예상 결과:** 메시지 형식 위반 제거

### 3단계: 검증 및 테스트 (1-2일)

- **조치:** 모든 이메일 전송 방법에 대한 종합적 테스트
- **예상 결과:** RFC 준수 문제 없는 일관된 전송

## 여전히 문제가 있나요?

RFC 5322 준수 문제가 계속 발생한다면:

- **구체적인 오류 문서화:** 정확한 반송 메시지를 수집하고 어떤 전송 방법이 문제를 일으켰는지 확인
- **체계적으로 테스트:** 캠페인, 워크플로우, 일괄 작업에서 각각 별도로 테스트 이메일 발송
- **최근 변경사항 검토:** 발신자 정보나 템플릿에 대한 수정사항 확인
- **설정 상호 참조:** 플랫폼의 모든 이메일 전송 방법에서 일관성 확인

### RFC 5322 준수 전문가 도움이 필요하세요?

RFC 준수 문제는 복잡할 수 있으며 이메일 표준에 대한 깊은 기술적 지식이 필요합니다.

다음과 같은 전문 지원을 받으세요:

- 고급 헤더 설정 및 검증
- 복잡한 MIME 구조 문제 해결
- GoHighLevel 이메일 설정 최적화
- 종합적인 템플릿 준수 점검
- 이메일 인프라 및 전송성 전략
